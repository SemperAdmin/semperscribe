# Document Upload & Metadata Extraction — Implementation Plan

## Concept

Let a user upload an existing Word document (`.docx`) or PDF that may not be formatted
correctly, extract its content in the browser, map the recognized pieces onto the app's
metadata fields, and drop the user into the normal editing flow — so they can export a
properly formatted document (SECNAV M-5216.5 compliant) from a mangled source.

Mangled doc in → review extracted fields → SemperScribe form filled → compliant PDF/DOCX out.

## Decisions (settled)

1. **Rules-only extraction.** No LLM, no network calls. Extraction runs entirely
   client-side, preserving the app's local-first / no-server-processing posture.
   Naval correspondence is rigidly conventional (SSIC, `From:`/`To:`/`Via:`,
   all-caps `Subj:`, `Ref: (a)`, `Encl: (1)`, `1.`/`a.`/`(1)` paragraphs), so a
   heuristic parser has a high ceiling here.
2. **Scope v1: the standard-letter family.** Basic Letter first-class, plus the
   memo types that share its field vocabulary (MFR, From-To Memo, Letterhead Memo).
   Endorsements, multiple-address letters, directives, forms, and AMHS come later.
3. **Import replaces the pending document.** No merging with in-progress work.
   The review modal warns that confirming will replace the current document.

## Non-goals (v1)

- Scanned/image-only PDFs (no text layer). Detected and reported to the user with a
  clear message; OCR (tesseract.js) is a possible later phase.
- Preserving rich formatting from the source.
- Perfect extraction. The review-and-confirm step is the contract: the parser produces
  a *draft*, the user verifies it.

## User flow

1. **File → Import Word/PDF Document…** in the File menu (`HeaderActions.tsx`).
2. User picks a `.docx` or `.pdf`. Text is extracted in-browser.
3. The **Review Imported Document** modal opens showing: detected document type
   (user can override, which re-runs the parse), every extracted field editable
   inline with low-confidence values flagged, editable Via/Ref/Encl/Copy-to/
   Distribution lists, a read-only paragraph preview, an "Unmatched text" section
   (nothing is silently dropped), and a warning that importing replaces the
   current document.
4. On **Confirm**, document state is reset (same reset as Clear Form) and the
   extracted data is applied through the existing `handleImport` path. Validation,
   live preview, and export all work immediately.
5. On **Cancel**, nothing changes.

## Architecture (all client-side)

```
src/services/import/
  documentTextExtractor.ts   # .docx / .pdf → ExtractedText (lazy-loaded libs)
  correspondenceParser.ts    # ExtractedText → ExtractionResult (pure, testable)
  docTypeDetector.ts         # ExtractedText → detected documentType + confidence
  extractionTypes.ts         # ExtractedText, ExtractedField, ExtractionResult, toImportPayload

src/components/import/
  DocumentImportModal.tsx    # review & confirm UI

src/hooks/
  useDocumentImport.ts       # orchestrates: file → extract → parse → modal state → apply
```

### Field semantics (important)

The service line (`UNITED STATES MARINE CORPS` / `DEPARTMENT OF THE NAVY` /
`DEFENSE LOGISTICS AGENCY`) maps to **`headerType`** (USMC/DON/DLA) — the app renders
it from that setting, it is not stored as a letterhead line. The letterhead lines
after it map by count to `line1` (unit name), optional `line1b` (unit sub-name,
flagged low-confidence because a four-line block is ambiguous), `line2` (street),
and `line3` (city/state/zip).

### Confidence

Every extracted field carries `confidence: 'high' | 'low'`, reusing the app's own
validators (`validateSSIC`, `validateSubject`, `validateFromTo`, `parseAndFormatDate`).
`low` values get an amber flag in the review modal. Source lines never claimed by any
rule are surfaced in the "Unmatched text" section.

## Dependencies

| Package | Why | Size discipline |
|---------|-----|-----------------|
| `mammoth` | docx → text in-browser | dynamic `import()` only inside `documentTextExtractor.ts` |
| `pdfjs-dist` | PDF text layer extraction | was already transitive via `react-pdf`; now a direct dep pinned to that same version (4.8.69); dynamic `import()`; worker configured the same way the app's react-pdf views do |

No server, no network calls, no telemetry — consistent with `SECURITY.md`.

## Testing (Vitest)

- `tests/services/import/correspondenceParser.test.ts` — clean, mangled, MFR,
  header-variant, sub-name, and closing-block fixtures; unmatched-text accounting.
- `tests/services/import/docTypeDetector.test.ts` — one fixture per supported and
  recognized-but-unsupported heading.
- `tests/services/import/documentTextExtractor.test.ts` — pdf line-assembly units,
  pdf-lib generated PDF fixtures (text layer, scanned, corrupt), and **round-trip
  tests**: a `.docx` generated by the app's own `docx-generator` is extracted and
  parsed, and the recovered fields must match the input.

- **Phase 2**: endorsements (parse "FIRST ENDORSEMENT on…" routing), multiple-address
